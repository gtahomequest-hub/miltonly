# Incident: ISR routes downgraded to lambda-only on /streets/[slug]

**Date:** 2026-05-25  
**Severity:** Medium (performance/cost, not correctness — pages 
rendered correctly but on every request)  
**Time to detect:** Unknown (Vercel deployed Node 24 default at 
some point; no monitoring caught the regression)  
**Time to resolve:** ~5 hours across two sessions  
**Resolved by:** Pin "engines": { "node": "20.x" } in package.json

## Symptom

/streets/[slug] returned `X-Vercel-Cache: MISS` and 
`Cache-Control: private, no-cache, no-store, max-age=0, 
must-revalidate` on every request to production, despite:

- Page exporting `revalidate = 3600`
- Page exporting `generateStaticParams` returning 291 slugs
- Build producing 292 .html prerender artifacts in 
  .next/server/app/streets/[slug]/
- prerender-manifest.json correctly listing each slug with 
  initialRevalidateSeconds: 3600
- Build route table showing ● SSG (not ƒ Dynamic)

The artifacts existed in the deployment but Vercel never served 
them. Every request hit a lambda.

## Root cause

Vercel project setting (Settings → General → Node.js Version) was 
set to 24.x. Next.js 14.2 + Vercel + Node 24 results in ISR routes 
being silently downgraded to lambda-only serving. `vercel inspect 
--json` showed `prerender: null` on the lambda entry for 
streets/[slug] despite a valid prerender-manifest entry existing.

No build error. No runtime error. No deprecation warning. The 
deployment simply ignored the prerender artifacts and served 
every request from a function with default no-store response 
headers.

## Hypotheses tried before finding the actual cause

Documented as anti-pattern reference. Each hypothesis sounded 
plausible and consumed time:

1. **Middleware setting headers** — eliminated by reading 
   src/middleware.ts; matcher narrowed to /streets/:path*, only 
   calls streetRedirect + NextResponse.next() with no header 
   mutation.

2. **next.config.js headers() block** — eliminated by reading 
   next.config.mjs; only contains redirects(), no headers().

3. **generateMetadata making uncached DB calls** — partially 
   eliminated; generateMetadata does call getStreetPageData but 
   that should be cacheable. Not the cause.

4. **Root layout poisoning every page via cookies()/headers() in 
   imported components (ChromeGate, UserProvider, 
   AttributionCapture)** — eliminated by reading each file; all 
   three are "use client" components, no server-side dynamic 
   calls. Grep across src/components confirmed no cookies(), 
   headers(), noStore, or connection() calls.

5. **Stale Vercel deployment / corrupted build cache** — would 
   have been a fallback hypothesis; never reached because actual 
   cause was found first.

## Why it was hard to find

Every hypothesis above was generated by inferring causes from 
response headers (`private, no-cache, no-store` + Vary: RSC 
headers). This is the wrong direction. The platform's view of 
the deployment (`vercel inspect --json`) was not consulted until 
hypothesis 5+, at which point the answer was obvious: `runtime: 
nodejs24.x` and `prerender: null` on a route that should have 
been prerendered.

The lesson: when build artifacts and runtime behavior contradict, 
consult the platform's manifest before generating more code-side 
theories.

## What would have caught it faster

1. A post-deploy verification script that runs `vercel inspect 
   --json` and asserts:
   - Node runtime matches package.json engines field
   - At least one expected-ISR route has prerender non-null
   - Cache headers on a known-static URL are `public, s-maxage=*`

2. A pinned `engines` field in package.json from project init, 
   so Vercel dashboard settings cannot drift the runtime without 
   a code change.

3. Initial debug evidence-gathering should always include 
   `vercel inspect --json | Select-String "runtime|prerender"` 
   when ISR symptoms are reported — before reading source code.

## Fix

Added `"engines": { "node": "20.x" }` to package.json. Committed, 
deployed. Verified:

- `vercel inspect --json` shows `nodeVersion: 20.x` and all 
  lambdas at `runtime: nodejs20.x`
- First curl on /streets/asleton-boulevard-milton: 
  `X-Vercel-Cache: PRERENDER`
- Second curl 6s later: `X-Vercel-Cache: HIT`, `Age: 6`
- `Cache-Control: public, max-age=0, must-revalidate`

## Side findings

Build log on Node 20 surfaced: `[cache] read failed for 
milton-sold-totals: Dynamic server usage: no-store fetch`. The 
Upstash Redis no-store fetches were tainting routes differently 
on Node 24 vs Node 20. Worth a follow-up audit of src/lib/cache.ts 
to confirm it behaves correctly on Node 20.
