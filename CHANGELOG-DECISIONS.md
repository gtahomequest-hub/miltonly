**3-layer GCLID + UTM attribution persistence — localStorage first-touch + last-touch + 90-day first-party cookie** (commit `7a13f5a`, Prisma migration `20260428223142_add_lead_attribution_persistence`) — Without persistence, `gclid` was dropped the moment a user navigated from `/rentals/ads` to `/listings/[mls]` before submitting; listing-detail submits got attribution-free leads, breaking Google Ads attribution. `captureAttribution()` runs on every page mount via `useSearchParams` re-trigger; first-touch is preserved in `localStorage.attribution` and a 90-day cookie (cookie acts as backup if storage is purged). Last-touch overwrites in `localStorage.attribution_last`. 12 form files spread `attributionPayload()` into their `/api/leads` body. Schema additions (all optional): `gclidLast`, `utmSourceLast/MediumLast/CampaignLast/TermLast/ContentLast`, `firstVisitAt`, `landingPage`. Existing `gclid` + `utm*` columns are now first-touch (no rename — preserves existing data) — Alternative: server-side session cookie attribution (rejected — Vercel Functions don't share session state across requests; client-side localStorage is the cheapest correct answer at this scale) or single-touch (rejected — Google Ads imports first-touch but our own multi-touch analytics need last-touch too).

**Manual Enhanced Conversions for Leads via SHA-256 hashed `user_data`, hashed client-side via `crypto.subtle`** (commits `e0c119a` → `ed43e49`) — Auto-detect Enhanced Conversions on Next.js multi-step forms reports near-zero match rate per Google's diagnostics. Manual hashing of email + phone (E.164) typically lifts match rate to 50%+. `src/lib/hash.ts` runs `sha256()` over `email.trim().toLowerCase()` and E.164-normalized phone; ThankYouClient fetches the lead by `lid` server-side (option B — keeps PII out of the URL) and passes email/phone props for client-side hashing. PII never reaches our telemetry — only the hash leaves the browser — Alternative for hashing location: server-side hash in `/api/leads` (rejected — would route raw PII through our logs even if redacted; client-side keeps PII at the source).

**Switch `user_data` forwarding to back-to-back `gtag('set','user_data',...)` + `gtag('event',...)` for GA4** (commit `ed43e49`) — The first implementation passed `user_data` inline as an event-param property: `gtag('event','generate_lead', { transaction_id, user_data })`. Inline `user_data` is the AW- (Google Ads native) pattern; gtag.js silently drops it when serializing to GA4's `/g/collect` endpoint because GA4 has no inline event-param slot for nested `user_data`. Verified by inspecting `/g/collect` payload showing `ep.transaction_id, epn.value, cu=CAD` but no `up.em` or `up.ph`. Canonical GA4 EC for Leads pattern is back-to-back set+event in the same synchronous tick — the queue is FIFO; the "race condition" warning about `gtag('set')` only applies when set is far separated in time from the event. Three call sites updated: ThankYouClient, fireGenerateLead in ListingExtras, submitLead in ListingDetailClient — Alternative: native AW- conversion path with `send_to: 'AW-…/conv_label'` (rejected — Google Ads new UI removed the manual gtag conversion path for new accounts; GA4-imported conversions are the only supported route).

**Resend auto-reply within ~30 seconds of lead submit, fire-and-forget from `/api/leads`, FROM hardcoded to `onboarding@resend.dev` until `miltonly.com` domain verifies** (commit `3a189d8`, refined `e968763` + `908cacf`) — Empirical data (Harvard study + real-estate benchmarks) shows leads contacted within 5 minutes are 8–10× more likely to convert than leads contacted within 60+ minutes. Email auto-reply within 30 seconds doesn't replace the call but bridges the gap. Fire-and-forget `resend.emails.send()` after `prisma.lead.create()` succeeds; errors caught + logged, never propagated to the client; skip when no email or honeypot. RECO compliance fix in `e968763`: brokerage corrected from `RE/MAX Real Estate Centre Inc.` (wrong) to `RE/MAX Realty Specialists Inc., Brokerage` in both HTML and plain-text versions; structured signature with Hall of Fame designation, compliance asterisk, tappable phone — Alternative: blocking send (rejected — adds 200–800ms to API response, breaks the speed-to-form-confirm promise) or wait for `miltonly.com` domain verification before launching auto-reply (rejected — domain verification is a separate workstream; sandbox sender lets the speed-to-lead promise ship Day 1).

**GA4 property "User-provided data collection" must be ON at admin level for Enhanced Conversions to forward `user_data` to `/g/collect`** (commit `41d9c5a` debug session, resolved in admin UI) — After fixing the inline-vs-set pattern bug (`ed43e49`), `/g/collect` payload still showed no `up.em`/`up.ph`. Five `[EC-DEBUG]` checkpoints confirmed the code chain was correct: server returns email+phone, client hashes them, gtag('set','user_data',…) fires before gtag('event','generate_lead',…). Root cause was the GA4 property: GA4 Admin → Data collection → "User-provided data collection" was OFF. The toggle is the property-level acknowledgement that GA4 is allowed to forward user_data to linked Ads accounts. Code path is necessary but not sufficient — both code AND admin toggle must be ON. Aamir flipped the toggle + Google Signals + granular location/device data. Logs stripped in `908cacf` once verified — Alternative: ignore at the property level and rely on code-path forwarding (impossible — GA4 explicitly drops user_data when the toggle is off; this is a GA4 design choice, not a bug).

**Vercel env var changes do not invalidate the build cache; explicit no-cache redeploy required for env changes that affect runtime** (debug session, AAMIR_EMAIL silent failure) — User reported `[twilio:stub] → aamir undefined` in production logs after adding `AAMIR_EMAIL=gtahomequest@gmail.com` to Vercel envs and triggering a redeploy. Investigation showed `process.env.AAMIR_EMAIL` was `undefined` at runtime despite the env being set in the Vercel UI. Cause: Vercel reused the previous build's compiled bundle, which had inlined the prior env values at build time. Fix: redeploy with "Use existing Build Cache" explicitly OFF in the Vercel UI, OR push an empty commit to trigger a fresh build. Diagnostic rule: read service logs literally — `[twilio:stub] → aamir undefined` shows `aamir` is the literal label and `undefined` is the value of `process.env.AAMIR_PHONE` (not AAMIR_EMAIL — separate env). Always verify the env name being read by the failing code path before changing the wrong env — Alternative: configure Vercel to bust cache on env changes automatically (not currently a Vercel feature; manual redeploy is the only path).

**Backfill script for historical `list_office_name` — AMPRE `or`-chain + www subdomain + escape-aware env loader** — `/api/sync/sold` is cursor-driven on `ModificationTimestamp > max_mod`, so the 7,001 sold_records rows that predated the `list_office_name` column were invisible to re-runs. ## 2026-04-16

**DB4 rule set rounded out — feature extraction, audit log, DB2 snapshots added** — Three rules added to the original set because they're load-bearing, not nice-to-haves. Feature extraction as a positive rule makes the "no raw records to Claude" prohibition enforceable against future "but this is for me personally" drift. Audit log (`_ingest_log`: source, timestamp, row count, hash, feature-version) turns append-only storage into audit-defensible storage — required the first time PROPTX asks for proof of non-redistribution. DB2-snapshot inputs (not live reads) preserve reproducibility when TREB issues corrections months later; without snapshots the forward-only schema and versioned feature columns collapse on the first revision. The full ruleset now survives Month 3 implementation without silent violations.

**Milton Intelligence DB (DB4) rules documented before build** — DB4 is scheduled for Month 3 but the rules governing it are locked now so Phase 3+ implementation respects them by construction. Core rules: (a) sold records only — no leases, no active listings; (b) append-only, never delete a row; (c) owner private use — raw records never consumer-facing, only derived features are; (d) fed from DB2 not directly from TREB; (e) schema forward-only, feature computation versioned; (f) raw records never enter Claude API. Reasoning: DB4's value depends entirely on historical completeness and immutability. If we ever delete a row we permanently lose training signal TREB may not re-expose; if we ever let a raw record render consumer-side we create VOW licence risk; if we mix sales + leases we poison the same "no lease data in ML" boundary we already enforced at the DB3 compute-function level. Documented in DO-NOT-REPEAT.md as settled rules — Alternative: wait until Month 3 and decide then (rejected — by then Phase 3 scoring pipeline may be built in a shape that silently violates one of these rules; cheaper to lock the rules while the code is still green-field).

**VOW agreement #1848370 activated; `VOW_TOKEN` provisioned and wired into sold routes** — IDX-scoped `TREB_API_TOKEN` returns zero rows for closed transactions because the IDX licence doesn't grant sold-data access. VOW is a separately-licensed feed. `/api/sync/sold` and `/api/sync/sold/test` now use `VOW_TOKEN`; `detect/route.ts` stays on IDX. Confirmed end-to-end path: VOW token + trimmed URL + `StandardStatus eq 'Closed'` filter returns real Milton closed records — Alternative: try to coerce IDX into returning sold data (rejected — licensing, not code).

**Capture both sales and leases in `sold.sold_records`, differentiated by `transaction_type` column** — `StandardStatus eq 'Closed'` returns both categories from VOW. Storing them in one table with a CHECK-constrained `transaction_type` column (`'For Sale'` or `'For Lease'`) lets one sync pipeline feed both — cheaper than parallel tables, indexed filter separates them at query time. Sales = intelligence + seller capture; leases = tenant/landlord capture, never touch the prediction engine — Alternative: two parallel tables `sales_records` + `lease_records` (rejected — doubles the sync pipeline and migration overhead without real isolation benefit).

**Two separate compute functions — `computeStreetSaleStats` and `computeStreetLeaseStats`** — Physical separation enforces "no lease data in ML" at the function boundary, not by application discipline. The sale function only selects `WHERE transaction_type = 'For Sale'`; the lease function only sees lease rows. DB4 imports the sale function and structurally cannot touch lease data — Alternative: single function with branching (rejected — mixing sale/lease filter logic in one function makes it easy to accidentally pipe lease data into sale metrics on a refactor).

**Lease analytics use bed-count breakdown, not a single average** — Rents vary ~3× by bed count (a 1-bed condo at $2,200 vs a 4-bed detached at $6,500). A single `avg_leased_price` mixing them is statistical noise. Analytics schema adds `avg_leased_price_1bed` through `avg_leased_price_4bed` plus an overall average for quick display. Sale prices cluster by property type already, so sale analytics retain their existing per-type breakdown — Alternative: single average (rejected — misleading for rent comparables).

**SQL `COMMENT ON` documenting dual-purpose table** — `sold.sold_records` holds leases too; table name becomes misleading. Rather than rename (high cost, many touchpoints), document at the SQL level: `COMMENT ON TABLE sold.sold_records IS 'Closed TREB VOW transactions — both sales and leases. ALWAYS filter on transaction_type before aggregating price data.'` Same treatment on `sold_price` column explaining its dual meaning. TypeScript doc comments on `SoldRecord` interface mirror the SQL — Alternative: rename table to `closed_transactions` (rejected — too many touchpoints for cosmetic clarity; comments + types are sufficient).

**Filter switch: `StandardStatus eq 'Closed'` instead of `MlsStatus eq 'Sold'`** — Confirmed via multi-probe diagnostic (`/api/sync/sold/test`): AMPRE's RESO-normalized `StandardStatus` uses `'Closed'` for all closed transactions regardless of sale/lease. `MlsStatus` uses `'Sold'` or `'Leased'` — filtering on `'Sold'` alone would miss leases. Using `StandardStatus` at the AMPRE boundary + `transaction_type` at the DB boundary gives one server-side filter that captures both categories cleanly — Alternative: two separate sync passes with `MlsStatus eq 'Sold'` and `MlsStatus eq 'Leased'` (rejected — doubles AMPRE round-trips for same result).

**Always `.trim()` TREB env vars at import** — `TREB_API_TOKEN`, `TREB_API_URL`, and `VOW_TOKEN` have all been observed with trailing whitespace/newline from Vercel env storage. Whitespace corrupts the URL query string and the Authorization header, producing a generic AMPRE 400 with no useful error. Every file that reads these env vars uses `(process.env.X || "").trim()` — Alternative: strip in a shared helper (deferred — three files is tolerable; revisit if it grows).

**Diagnostic endpoint pattern: always return HTTP 200** — `/api/sync/sold/test` runs multiple AMPRE probes in parallel and returns HTTP 200 unconditionally. Success of each upstream call is conveyed in the JSON body (`ok`, `amprerror`, per-probe `httpStatus`, distinct `MlsStatus`/`StandardStatus`/`TransactionType` arrays). Returning non-200 from a diagnostic endpoint makes it look crashed when it worked correctly — we went through exactly that trap earlier in the session. Pattern now codified — Alternative: pass through AMPRE's status code (rejected — collapses diagnostic signal into HTTP noise).

**Cursor pagination uses `ModificationTimestamp`, not `CloseDate`** — `CloseDate` is `Edm.Date` in AMPRE and rejected in `$filter` with "Field not allowed in filter: CloseDate". `ModificationTimestamp` is `Edm.DateTimeOffset` and both filterable and orderable. Both backfill and incremental modes now paginate on the same field with different initial cursor states; simpler and works for mixed sale/lease result sets — Alternative: filter by `PurchaseContractDate` (untested; deferred).

**Hand-rolled SQL statement splitter in migration runner** — Neon's HTTP transport accepts one statement per call. Migration files contain many statements separated by semicolons.
